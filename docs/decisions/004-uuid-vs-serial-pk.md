# 004. UUID primary keys instead of SERIAL

## Context
Every table needs a PK. The default in Postgres is SERIAL (auto-increment int).
We went with UUID, generated by the database via gen_random_uuid().

## Decision
UUID PKs, generated by Postgres (server_default), not by Python.

## Why
- SERIAL leaks information. /books/1 tells you it's the first book ever created.
  UUIDs don't expose count or order.
- Generating in the DB means one source of truth. With bulk insert, the app
  doesn't need to know IDs ahead of time — Postgres assigns them and we get
  them back via RETURNING.
- Safe if we ever merge data from multiple sources. No ID collisions.

## Cost we accept
- 16 bytes vs 4-8 for an int. Irrelevant at our scale.
- UUIDs don't sort by creation time. We don't need them to — that's what
  created_at is for.
